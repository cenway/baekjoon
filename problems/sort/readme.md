일단 코테 기준으로, stl가능 => sort(), 불가능=> 머지 소트

STL sort를 쓸 때
1. 항상 끝 + 1 위치(이건 stl 대부분)
2. pair, tuple 사용 시 알아서 앞에서부터 비교
3. 정렬 함수 전달 시 bool cmp(a,b);
4. return a > b;(a == b 이면 false) :::내림차순
5. 인자는 가급적 레퍼런스로 전달(복사를 막기 위함)
ㄴ나중에 C로 코테 볼 땐 qsort 사용법을 꼭 따로 익혀둘 것

stable sort의 개념을 기억하고 있을 것.
ㄴ여러 번 정렬해야 할 경우, 정렬 속에 정렬이 필요할 경우
ㄴstl의 경우 stable_sort를 제공함

퀵 소트의 경우 cache hit rate도 높아서 평균적인 속도가 머지 소트보다 빠름
또, 머지 소트처럼 추가적인 공간도 필요가 없음(In-Place sort)
그러나 퀵 소트는 최악의 경우, 즉 분열이 안 되고 거의 대부분이 순차적으로 놓여 있고
pivot을 왼쪽에서만 잡는 경우(1 2 3 4 5 6 7 8) O(N^2)의 시간 복잡도가 나옴

보통 라이브러리에선 pivot을 잡는 방식이 다르거나 일정 깊이 이상이되면 힙소트로 바꾸는 등의 처리가 있음
그러나 개인이 이까지 구현하는 건 코테 시간 상 부적절

그리고 퀵소트는 unstable sort임

대상 값의 범위가 작고, 단순한 수처럼 대상이 unique하지 않으면 카운팅 소트가 좋은 선택이 될 수 있음

라딕스 소트는 개념만 이해하고 넘어가기
ㄴstl없이 구현하면 실수하기 쉽고, stl이 있으면 sort()
